Facts and Questions

Facts and questions in AGISystem2 are expressed exclusively through Sys2DSL, not as “free” text lines. Every interaction is a triple statement of the form @var Subject VERB Object inside a session or a theory file.

Facts in Sys2DSL (Triple Assertions)

A fact is a Sys2DSL instruction that binds a variable to a canonical triple Subject VERB Object and updates the active theory. There is no special ASSERT command in v3.0 – you simply use verbs such as IS_A, HAS, CAUSES, etc.

General form:

@_ Subject VERB Object

Examples of basic facts:

@_ Dog IS_A animal
@_ Water HAS boiling_point
@_ Water BOILS_AT Celsius100
@_ surgery_x REQUIRES consent
@_ export_data REGULATED_BY GDPR

Note: Properties like boiling point are expressed using specific verbs (for example BOILS_AT) that connect subjects to value concepts (for example Celsius100). This keeps values as first-class concepts that can be reasoned about independently.

Conventions for tokens in the parameter list:

  • a word starting with a lowercase letter (animal, water, export_data) is treated as a concept name;
  • a word starting with an uppercase letter (Dog, Alice, CityX) is treated as an individual or concrete instance (a fact point);
  • a word starting with $ ($factList) is a reference to a variable already defined in the script.

Any triple statement that uses verbs like IS_A, HAS, CAUSES, etc. both creates/updates points in conceptual space and returns a result point. There is no separate “assert vs query” syntax – that distinction is semantic, not syntactic.

Yes/No Questions (Triple Queries)

Questions are also Sys2DSL instructions. They bind a variable to a result point whose existence dimension encodes the truth value. The form is still:

@q1 Subject VERB Object

Typical yes/no queries use the same verbs as assertions:

@q1 Dog IS_A animal
@q2 Water BOILS_AT Celsius100
@q3 export_data REGULATED_BY GDPR

To enumerate or “ask what/which”, use the any wildcard and dedicated query verbs from the v3.0 spec:

@animals any IS_A animal        # All animals
@dogFacts Dog FACTS any          # All facts about Dog
@causes any CAUSES smoke         # What causes smoke?
@result Dog QUERY animal         # High-level query verb

In all cases, the result is stored in the variable (for example @q1) and can be read or combined by other instructions. See the design specs for IS_A, FACTS and QUERY in the Sys2DSL_commands* documents for exact semantics.

Counterfactual Questions (CF)

Counterfactuals can be expressed either via high-level verbs like WHATIF or via the CF verb described in the core command spec. In both cases we still use triple syntax.

High-level example (see Sys2DSL_highlevel.md):

@result Water WHATIF gas

Core CF form from DS(/theory/dsl_commands_core.js):

@result Query CF "Water IS_A gas? | Water BOILS_AT Celsius50"

Here Query is the subject, CF is the verb, and the object is a structured string that contains a question and a list of hypothetical facts separated by | and ;. Internally, the engine uses theory layers so that these assumptions do not modify the base theory.

Combining Facts and Questions in One Script

A typical Sys2DSL script that mixes facts and questions looks like this:

@_ Dog IS_A animal
@_ Water BOILS_AT Celsius100
@_ Celsius100 IS_A temperature
@q1 Dog IS_A animal
@q2 Water BOILS_AT Celsius100

You can write the same instructions on a single line, for example:

@_ Dog IS_A animal @q1 Dog IS_A animal

The interpreter splits the script at each @, builds the dependency graph between variables and executes the instructions in a topological order so that $ references are resolved correctly even if they appear “earlier” in the text.

Summary

In v3.0 triple syntax, both facts and questions use the unified form @var Subject VERB Object. There is no separate ASSERT/ASK command layer; instead, verbs like IS_A, HAS, FACTS, QUERY, WHATIF, etc. all follow the same triple pattern. This guarantees a single, analyzable interaction form that is easy to compose into theory programmes.
