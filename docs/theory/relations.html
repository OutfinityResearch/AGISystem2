Relations: The Grammar of Structure

Relations are the verbs of the conceptual space. Each relation is a deterministic permutation that encodes a role without adding dimensions. We start with an OWL-inspired set to cover most structural, causal, temporal, and deontic needs, and we allow adding new relations when domains demand it.

Default Set (OWL-inspired)

Structural: IS_A (subClassOf, transitive), EQUIVALENT_TO (symmetric), DISJOINT_WITH (symmetric), PART_OF/HAS_PART (transitive), HAS_PROPERTY/PROPERTY_OF (functional/inverse-functional), LOCATED_IN/CONTAINS.

Causal/Temporal: CAUSES/CAUSED_BY, BEFORE/AFTER (transitive), AGENT_OF/HAS_AGENT, PATIENT_OF/HAS_PATIENT.

Deontic: PERMITS/PERMITTED_BY, OBLIGATES/OBLIGATED_BY, PROHIBITS/PROHIBITED_BY.

Fallback: RELATED_TO (symmetric, low weight) when nothing else fits.

How Relations Become Geometry

Each relation gets a permutation table; symmetric relations reuse the same table; inverse pairs get paired tables. Binding is a permutation followed by addition; unbinding (for abduction) uses the inverse permutation. Because permutations are seeded, the same relation always yields the same shuffle across runs.

Adding New Relations Safely

1) Define the intent: is it structural, causal, temporal, deontic, or a domain-specific role? 2) Decide symmetry and transitivity. 3) Register the relation with the permuter using a seed from config; if it has an inverse, register both together. 4) Update the grammar and fixtures so ingestion accepts it. 5) If the relation implies constraints (e.g., functional), document them in the domain theory.

Domain Hints

Relations can bias ontology or axiology axes. For example, PERMITS/PROHIBITS primarily touch axiology, while IS_A/PART_OF stress ontology. Temporal relations can be paired with TemporalMemory rotations for sequence modeling.

Example binding:
  "Fire CAUSES Smoke"
  fireVec + permute(smokeVec, CAUSES)

Example abduction:
  observation = smokeVec
  hypothesis = inversePermute(observation, CAUSES_inverse)
  nearest(hypothesis) -> Fire

Keep the relation set small unless you have a clear semantic need; every new relation is a new permutation to maintain and test.
